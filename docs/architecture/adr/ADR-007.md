# ADR-007: Body-Based Refresh Token Delivery

Status: Frozen for MVP implementation

## Status

Accepted

## Context

The MVP needs login, refresh, and logout flows that work reliably through the Nginx edge in local Docker Compose and on a single VPS. Refresh token delivery must align with the OpenAPI contract and avoid hidden browser cookie behavior during the first implementation pass.

## Decision

Use JSON body delivery for refresh tokens in MVP.

`AuthResponse` returns the refresh token in the JSON response body. `RefreshRequest` and logout requests submit the refresh token in the JSON request body. Refresh tokens remain opaque random values and are stored only as hashes in PostgreSQL.

Cookie-based refresh token delivery with `httpOnly`, `Secure`, and `SameSite` attributes is deferred as a post-MVP hardening option.

## Options Considered

| Option | Pros | Cons |
|---|---|---|
| JSON body refresh token | Matches current OpenAPI contract, simple local/VPS CORS setup, easy to test with Postman and frontend state | Frontend must protect token storage carefully |
| httpOnly cookie refresh token | Stronger browser storage boundary | Requires cookie-specific CORS, SameSite, domain, HTTPS, and local development handling |
| Access token only | Simplest auth implementation | Weak session continuity and no refresh rotation |

## Consequences

- Frontend auth state owns both access token and refresh token for MVP.
- Backend refresh and logout endpoints validate refresh tokens from JSON bodies.
- CORS does not need cookie credentials for refresh-token behavior.
- Security docs and API contract must clearly state the body-based JSON request and response behavior.
