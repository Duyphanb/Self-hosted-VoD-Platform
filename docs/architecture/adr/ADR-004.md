# ADR-004: MinIO For MVP Object Storage

Status: Frozen for MVP implementation

## Status

Accepted

## Context

The platform must store large raw video files and generated HLS artifacts. Local filesystem storage inside app containers would be fragile and hard to migrate.

## Decision

Use MinIO as the MVP S3-compatible object storage layer.

Raw uploads, HLS artifacts, and future thumbnails use separate buckets. Optional AWS S3/CloudFront can be introduced later through an adapter without changing core business flows.

## Options Considered

| Option | Pros | Cons |
|---|---|---|
| MinIO | Local and VPS friendly, S3-compatible, portfolio-relevant | Must manage buckets and credentials |
| Local filesystem | Very simple | Poor portability, weak production resemblance |
| AWS S3 first | Production-like | Adds cloud cost and setup before MVP stability |

## Consequences

- Object keys are stored in PostgreSQL.
- Buckets are private by default.
- Backend and worker need MinIO configuration.
- HLS objects stay private and are served only through Nginx `/hls/*` to the backend HLS proxy.
- Future S3 migration should preserve the same storage interface.
