# ADR-001: Modular Monolith Plus Worker

Status: Frozen for MVP implementation

## Status

Accepted

## Context

The project needs to demonstrate real backend and full-stack engineering while remaining feasible for a solo developer. The system includes user APIs, catalog management, uploads, asynchronous video processing, and playback.

## Decision

Use a modular monolith for core business APIs and a separate worker runtime for media processing.

The backend owns user-facing APIs and business workflows. The worker owns FFprobe, FFmpeg, HLS generation, and processing status updates.

## Options Considered

| Option | Pros | Cons |
|---|---|---|
| Single Spring Boot app does everything | Simplest deployment | Upload requests and media processing become tightly coupled |
| Modular monolith plus worker | Clear responsibility split, still operationally simple | Requires queue and shared contracts |
| Microservices | Strong service isolation | Too much operational complexity for MVP |

## Consequences

- The backend remains the source of API truth.
- The worker can fail or restart without taking down normal browsing APIs.
- Queue-based media processing is required.
- Service boundaries must stay disciplined to avoid accidental microservice scope.
