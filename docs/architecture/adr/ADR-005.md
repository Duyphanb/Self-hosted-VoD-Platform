# ADR-005: RabbitMQ For Encoding Job Queue

Status: Frozen for MVP implementation

## Status

Accepted

## Context

Video encoding is slow and resource-intensive. Upload requests must not block until FFmpeg completes.

## Decision

Use RabbitMQ as the asynchronous queue between backend and worker.

The backend publishes encoding jobs after raw upload succeeds. The worker consumes from `video-encoding-queue` and updates PostgreSQL status.

## Options Considered

| Option | Pros | Cons |
|---|---|---|
| RabbitMQ | Reliable queue semantics, easy local Compose setup | Adds broker dependency |
| Direct in-process async | Simpler short term | API process owns media workload and failure scope |
| Kafka | Strong event platform | Too heavy for MVP and explicitly out of scope |
| Database polling only | Fewer services | Less explicit queue behavior and weaker portfolio signal |

## Consequences

- Encoding job payloads need stable contracts.
- Failed jobs must be visible in PostgreSQL.
- Worker concurrency defaults to 1.
- DLQ configuration is included in the queue setup so failed or poison messages remain visible.
- Advanced retry policies are deferred until after MVP stability.
