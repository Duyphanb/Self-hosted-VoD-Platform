# ADR-006: HLS With FFmpeg And FFprobe

Status: Frozen for MVP implementation

## Status

Accepted

## Context

The MVP requires browser playback of uploaded videos without manual file handling. The system should demonstrate real media processing while staying achievable on a single VPS.

## Decision

Use FFprobe for metadata extraction and FFmpeg for HLS generation.

The MVP targets a single HLS output path. Multi-quality adaptive bitrate is deferred until the single-quality upload -> transcode -> playback flow is stable.

Browser playback requests HLS only through Nginx `/hls/*`, which forwards to the backend HLS proxy. Direct MinIO, public bucket, signed URL, and presigned playback URL delivery are not part of MVP.

## Options Considered

| Option | Pros | Cons |
|---|---|---|
| FFmpeg HLS | Standard toolchain, strong portfolio value, browser-friendly with HLS.js/Video.js | Requires careful worker and VPS testing |
| Serve raw MP4 only | Simpler | Does not meet the intended streaming architecture |
| Managed transcoding service | Operationally easier | Not self-hosted and adds cloud dependency |
| Multi-quality HLS immediately | More production-like | Too much scope before MVP stability |

## Consequences

- Worker image must include FFmpeg and FFprobe in `PATH`.
- HLS MIME types must be configured in Nginx.
- Encoding correctness is more important than speed for MVP.
- Resource usage must be watched on Oracle VPS.
